1 · The problem as physics

Strip away the product. The underlying situation is: a machine makes thousands of near-identical decisions per hour, and a single upstream mistake corrupts all of them silently.

A campaign dials thousands of people. Each call pulls that person's data (name, EMI, due date) and speaks it. If the mapping from the uploaded list to the agent breaks, the agent doesn't crash — it just uses a default value for everyone. Every call still "succeeds." The dashboards still look normal. The only symptom is semantic: the agent says "am I speaking with <the same default name>?" on every call.

why it's hard Nothing errors. There's no exception, no 500, no failed job. The failure is invisible to conventional monitoring because it lives in the content, not the status. That's why a human had to randomly eyeball Metabase to find it — and why it ran for 3–4 days.

So "catching it" requires monitoring a different thing than uptime: you must watch the distribution of the data flowing through, and notice when it deviates from what "healthy" looks like. That reframing — monitor the data, not the process — is the seed of the entire system.

2 · The domain model from zero

Five nouns, nested. Hold these and everything else clicks:

NounWhat it really is
OrgA client tenant (tatacapital-housing.com). Everything is scoped by org_id. Multi-tenant from day one.
App / agentThe AI that talks. Has app_id + app_version. The "program" being run.
CampaignA scheduled dialing job: this list, this agent, these hours, this retry policy. Has a status lifecycle: scheduled → active → ended.
CohortThe uploaded contact list (CSV). Validated into total / valid / rejected records. The data source for per-call variables.
Agent variablesThe per-call values (customer_name, emi_amount, due_date). Some are per-contact (differ every call); some are campaign config (same for everyone: bot_name, gst_rate).
the crux The bug lives in the seam between cohort and agent variables: a cohort_transformation maps CSV columns → variables. When that mapping silently fails, the per-contact variable falls back to a default, and the collapse begins. Every detector is, in some sense, watching that seam.

3 · Where truth lives (and the encryption wall)

All data is reached through Metabase's REST API (native SQL). We picked Metabase's API key deliberately: it works headless — from a VM or a Kubernetes pod, no browser, no login. (The alternative, the samvaad shell helper, scrapes a cookie from a running Chrome — laptop-bound, useless on a server. That's why we never used it.)

StoreEngineHoldsUsed for
DB 13 EngagementFactsClickHouseone row per call: connectivity, completion, durations, on_start/on_end_agent_variables (Maps)detectors, cycle report
DB 6 schedulingPostgrescampaigns, cohorts, cohort_transformationsdiscovery, run summary, required-var hints
DB 10 insights_resultPostgresper-call eval metrics: hallucination, loop_detection, call_effectiveness…weekly evals, quality detectors
the wall you must remember Transcript text is encrypted at rest. Both InteractionMessages.content and EngagementFacts.on_end_interaction_transcript are base64 of <keyid>|||<org>|||<AES ciphertext>. Structured fields (variables, dispositions, metrics) are plaintext; the words spoken are not. This single fact decides what's possible: anything needing to read the conversation is off the table from Metabase. We only learned this by decoding a sample — see §8.

4 · The one idea the whole system rests on

The naive detector is "a variable with the same value everywhere is a bug." This is wrong, and the data proves it: bot_name=नेहा, gst_rate=18, campaign_code=CMP0526 are supposed to be identical across every call. Flagging low cardinality would drown you in false positives.

the real signal The bug is not "low cardinality." It's a variable that is normally per-contact suddenly collapsing to one value. "Normal" is defined by the campaign's own recent history. So every detector compares a current window (last 6h) against a baseline window (prior 72h) for the same campaign — never against an absolute threshold.

Two independent ways to know a variable "should" be per-contact:

  • History: in the baseline it had many distinct values (e.g. customer_name had 386,690). If it drops to 1 now → bug.
  • Intent: the cohort transformation marks it required — it must come from the upload. A required variable that collapses is high-confidence, and this works even on a brand-new campaign with no history yet (the "catch it at launch" case).

Internalize this and you understand 80% of the codebase. Everything is "current vs baseline, per campaign, with intent hints."

5 · The pipeline shape

discover active campaigns (cheap: scheduling DB) │ ▼ for each campaign build a snapshot context ──► run detectors (ClickHouse aggregation) │ │ findings ▼ ▼ build reports dedupe / cooldown (SQLite/S3) │ reports │ new findings └──────────► notifiers ◄───────┘ console · Slack webhook · Slack bot · Google Sheet

Two design choices worth understanding:

  • Discovery is cheap on purpose. It lists active campaigns from the small scheduling DB, then does one bounded campaign_id IN (…) count query on the giant facts table. The earlier version scanned the whole facts table grouped by campaign and timed out (504) at the gateway. Lesson: never scan the big table unfiltered.
  • Detectors are pure-ish functions. Each takes a context (campaign + config + Metabase client) and returns Findings. They never notify, never touch state. That separation is why you can add one in ~40 lines (see §12).

7 · Alerts vs reports: two streams

Not everything is an alarm. The system has two output streams, and a notifier subscribes to one or both:

  • alerts — severity findings from detectors, deduped (you get pinged when a problem appears, not every 30 min).
  • reports — always-posted digests: the run summary (cohorts, rows uploaded vs filtered, retry windows — "what is the automation doing right now"), the cycle report (connectivity/engagement/PTP/disposition funnel), weekly evals (insights quality per campaign), and the client report (per-org markdown deliverable).
why split Alerts must be rare and actionable (alarm fatigue kills alerting systems). Reports are ambient situational awareness. Mixing them destroys both. The streams concept keeps them apart while sharing the same delivery plumbing.

8 · The LLM saga (a lesson in verifying)

This is the most instructive arc in the project — keep it as a permanent reminder to verify assumptions against reality.

  1. We built Layer A (metric-based conversationality: looping/silence/no-closure from plaintext flags) and Layer B (feed transcripts to gpt-4o/mini, score quality + check "was the EMI stated correctly").
  2. We calibrated mini vs gpt-4o (mini ran ~0.9 pts harsher; picked gpt-4o). Felt rigorous.
  3. Verifying on real campaigns, every result was suspiciously flat: always early_drop, always "not mentioned."
  4. We decoded a transcript. It was ciphertext. The LLM had been scoring base64(AES(...)) the entire time. The calibration numbers were noise-on-noise.
the lesson A pipeline can be fully "working" — tests green, calls succeeding, plausible-looking output — and be completely meaningless because an upstream assumption (transcripts are readable) was false. The fix wasn't more code; it was checking the input. Now those features have an encryption guard (detect ciphertext → skip with a clear message) and conversation quality comes from the insights_* tables, which are decrypted upstream.

9 · Delivery, dedupe, and ephemerality

  • Notifiers implement one interface: console (rich terminal / task logs), slack_webhook (one channel per URL), slack_bot (any channel, per-campaign threads, per-org routing), gsheet (auto-fills the Master QC sheet). Deep links + example interaction_ids ride along so a human jumps alert → evidence.
  • Dedupe is a SQLite table of finding fingerprints + last-notified time. A finding re-alerts only after cooldown_hours. This is what makes alerts rare.
ephemerality On Kubernetes each run is a fresh pod — local SQLite doesn't survive. So the scan syncs its dedupe DB to/from S3 around each run. Miss this and you'd re-alert every single pass. Ephemerality quietly breaks any "remember across runs" assumption; the fix is always "externalize the state."

10 · Deployment: why Airflow changes the design

Originally this targeted a VM (systemd timer, long-running watch loop, local state). Then the platform moved to Airflow/Kubernetes, and that migration forced three changes — a good example of how infra shapes code:

VM assumptionK8s realityConsequence
always-on processephemeral pods per taskstate → S3; jobs → DAGs on cron schedules
secrets in .envK8s Secretinjected via pod_override env, never committed
local depscustom imagepackage installed from git into images/samvaad, config baked in

Six DAGs (samvaad_alerting_scan every 30 min + daily/weekly report DAGs) run the exact same library code the CLI runs — the CLI and DAGs are two front-ends over one engine.

11 · Hard-won lessons (the scar tissue)

  • 504 on wide scans → never GROUP BY the full facts table; always filter by campaign_id IN (…) first. Discovery moved to the small scheduling DB.
  • Encrypted transcripts → structured data is plaintext, words are not; use the insights pipeline for anything semantic.
  • "Same value = bug" is false → baseline-relative detection is mandatory (§4).
  • LLM sampling caught only early-drops → prefer completed/longest calls, or you evaluate hangups.
  • Insights aren't daily-dense → the eval lookback is 7 days, not 3, or campaigns show no data.
  • Metabase returns HTTP 202 (not 200) on success from /api/dataset — the client accepts both.

12 · How to extend it without fear

Adding a detector is a closed, mechanical task — this is the payoff of the pure-function design:

1. new file in detectors/, subclass Detector, set name = "my_detector"
2. run(ctx) -> list[Finding]     # one SQL via ctx.metabase.query(...)
3. register in detectors/__init__.py DETECTOR_CLASSES
4. add [detectors.my_detector] to config (enabled + thresholds)
5. it now runs in every scan; findings flow through dedupe + notifiers free

You never touch delivery, dedupe, discovery, or the DAGs. That's the whole point of the seams.
